# NordicBlocks — Catalog Family V1

Дата: 2026-04-19

## 1. Зачем нужна эта линейка

После news/editorial направления следующей крупной продуктовой задачей для NordicBlocks становится каталожная линейка.

Её задача:

1. показывать работы, услуги, товары и кейсы в виде управляемого mini-catalog на лендинге;
2. поддерживать быстрый сценарий без корзины, заказа и оплаты;
3. давать живой поиск, фильтры и сортировку внутри одной секции;
4. уметь работать и в `manual`, и в `content_list` режиме через уже принятый InstantCMS data pipeline;
5. выглядеть как продуктовая мини-система, а не как обычный repeater block.

Главный product-смысл:

это не магазин и не полноценный ecommerce-модуль, а `commerce-lite` / `catalog-explorer` слой для лендингов.

## 2. Почему это уже не просто один блок

Для этой задачи одного изолированного блока недостаточно, потому что здесь появляются связанные product-capabilities:

1. toolbar с поиском и фильтрами;
2. единая модель карточек каталога;
3. категория как навигационная сущность, а не только маленькая метка;
4. price/filter state;
5. единый способ подключения данных из InstantCMS;
6. одинаковая логика preview/live.

Поэтому правильная модель такая:

1. продуктово это `мини-компонент`;
2. архитектурно это `семейство из 3 родственных block types`;
3. технически всё семейство должно жить на одном shared catalog runtime.

## 3. Product-first позиционирование

Эта линейка нужна для трёх типовых сценариев лендингов:

1. `Портфолио / работы`
   Показ фото работ компании с категориями, тегами и быстрой фильтрацией.
2. `Услуги по категориям`
   Категории услуг, внутри которых лежат карточки с описанием, ценой от и CTA в мессенджер.
3. `Мини каталог товаров`
   Небольшой магазин без checkout: только просмотр, фильтры, цена, кнопка связи или переход в мессенджер.

Общее правило:

блоки этой линейки отвечают за discovery и lead generation, а не за оформление заказа.

Практический референс, на который можно опираться при проектировании этой линии:

1. в проекте `nordic-builder.ru` уже существуют блоки `mini-shop-catalog-01` и `cases-catalog-01`;
2. `mini-shop-catalog-01` подтверждает правильную toolbar/filter/search модель для commerce-lite каталога;
3. `cases-catalog-01` подтверждает правильную модель popup/modal navigation для карточек и галереи;
4. при проектировании NordicBlocks-линии не копируем их буквально, но считаем их валидными продуктовыми ориентирами.

## 4. Family-first состав линейки

Первая мини-линейка фиксируется так:

1. `Каталог`
2. `Категории и каталог`
3. `Витрина каталога`

Роли внутри линейки:

1. `Каталог` — базовый system-forming block с поиском, фильтрами и grid карточек;
2. `Категории и каталог` — category-first вариант, где навигация по категориям является частью композиции;
3. `Витрина каталога` — более акцентная версия для лендинга: featured category / featured item + фильтруемая выдача.

## 5. Первый блок к реализации

Первым блоком линейки должен стать:

1. пользовательское название: `Каталог`;
2. рабочий технический slug рекомендуется держать нейтральным, например `catalog_browser`;
3. этот блок обязан стать базой для card contract, filter model и toolbar runtime;
4. остальные два блока должны строиться уже поверх той же catalog engine.

## 6. Главная идея v1

В v1 делаем не «всё сразу», а один сильный универсальный catalog block, который уже умеет:

1. выводить карточки;
2. искать по названию и описанию;
3. фильтровать по категориям;
4. фильтровать по цене;
5. сортировать;
6. переключаться между manual и dynamic source;
7. вести пользователя на внешний URL или в мессенджер.

## 7. Канонические режимы данных

Для семейства фиксируем два режима v1:

1. `manual`
2. `content_list`

### `manual`

Используется когда:

1. нужно быстро собрать каталог руками;
2. это демо-сценарий для блока;
3. данные ещё не заведены в InstantCMS;
4. лендинг небольшой и управляется вручную.

### `content_list`

Используется когда:

1. карточки должны приходить из выбранного content type InstantCMS;
2. каталог должен обновляться без ручного дублирования;
3. нужно брать услуги, товары, проекты, работы или кейсы из реальных данных.

Важное правило v1:

1. runtime получает уже подготовленный payload через существующий data pipeline;
2. `render.php` не должен сам ходить в базу;
3. preview и live обязаны использовать один и тот же hydration/merge flow.

## 8. Shared catalog engine

Для этой линейки нужен общий слой, который обслуживает все block types family.

Условное название:

`catalog engine`

Что в него входит:

1. единый contract shape для catalog items;
2. toolbar state model;
3. filter model;
4. client-side search/filter runtime;
5. data mapping из `content_list`;
6. единая карточка каталога и её сущности;
7. graceful empty state;
8. единые правила кэша и identity.

Практическое решение для v1:

1. сервер подготавливает список элементов через существующий `content_list` adapter;
2. toolbar search/filter/sort работают на фронте по уже загруженному payload;
3. это даёт симметрию preview/live и не требует сразу remote faceting backend;
4. remote query mode, pagination и большие каталоги уходят в следующий этап.

## 9. Что должен уметь первый блок `Каталог`

### 9.1 Сценарии использования

1. каталог работ с фото и категориями;
2. каталог услуг с ценой и кнопкой `Написать`;
3. мини-каталог товаров без заказа;
4. подборка кейсов по направлениям;
5. каталог меню/прайса для локального бизнеса.

### 9.2 Базовый UI v1

Секция состоит из:

1. header секции;
2. toolbar поиска и фильтров;
3. active filters summary;
4. grid карточек;
5. empty state;
6. optional more link / CTA секции.

### 9.3 Медиа-поведение v1

Для этой линейки фиксируем отдельное обязательное правило по изображениям карточек:

1. клик по фото карточки должен уметь открывать media modal;
2. для phone-first сценария целевой режим по умолчанию: `fullscreen`;
3. modal обязан поддерживать удобное перелистывание соседних элементов;
4. на touch-устройствах перелистывание должно работать жестом swipe;
5. на desktop должны быть явные prev/next controls;
6. modal не должен ломать scroll и focus-management страницы.

Практический product-смысл:

1. для каталога работ это превращает блок в удобный portfolio viewer;
2. для услуг и товаров это даёт быстрый zoom/detail сценарий без перехода на отдельную страницу;
3. это повышает ценность блока как mini-component, а не просто grid карточек.

## 11. Contract shape для catalog item

Каждая карточка v1 должна уметь хранить:

1. `category`
2. `category_url`
3. `title`
4. `excerpt`
5. `image`
6. `imageAlt`
7. `price`
8. `price_old`
9. `currency`
10. `badge`
11. `tags[]`
12. `url`
13. `cta_label`
14. `cta_kind`
15. `cta_url`
16. `messenger_type`
17. `availability`
18. `gallery[]`
19. `sku` или внутренний код при необходимости

Где `gallery[]` поддерживает:

1. `src`
2. `alt`
3. `type` (`image` как минимум в v1)
4. `caption` при необходимости

Замечание:

не все поля обязаны быть видимыми одновременно, но contract должен поддерживать их как единый catalog language.

### Density preset

Для этой линейки нужен отдельный контроль плотности.

Канон:

1. дефолтный preset — `dense`;
2. `dense` должен наследовать общий NordicBlocks курс на плотную grid-подачу;
3. для базового dense режима допустим `gap: 0` или визуально почти нулевой micro-gap;
4. воздушный каталог не должен быть стартовым дефолтом.

## 16. Filter model v1

В v1 фиксируем только те фильтры, которые реально дадут сильный продукт без giant-engine.

### 16.1 Search

Поиск работает по:

1. `title`
2. `excerpt`
3. `category`
4. `tags`

### 16.2 Category filter

1. flat category list;
2. single-select в базовом сценарии;
3. multi-select можно закладывать в contract, но не делать обязательным для первого UI.

### 16.3 Price filter

1. `min`;
2. `max`;
3. работает только если у карточек реально есть numeric price.

### 16.4 Sort options

1. manual order;
2. title asc;
3. title desc;
4. price asc;
5. price desc;
6. newest, если источник поддерживает date.

## 17. Media modal / gallery model v1

Для этой family v1 media modal считается частью канона, а не опциональным украшением.

### 17.1 Целевое поведение

1. клик по cover image открывает modal viewer;
2. если у карточки есть `gallery[]`, modal стартует с cover и позволяет идти дальше по изображениям;
3. если у карточки только одно изображение, modal работает как single image viewer;
4. для mobile fullscreen режим является дефолтным;
5. для desktop допустим fullscreen или centered, но fullscreen остаётся предпочтительным для portfolio/cases сценариев.

### 17.2 Navigation

1. prev/next controls обязательны;
2. keyboard left/right желательны для desktop runtime;
3. swipe left/right обязателен для touch runtime;
4. close button должен быть всегда легко доступен на телефоне;
5. background tap для close допустим, но не заменяет явную кнопку закрытия.

### 17.3 Что показывается внутри modal v1

1. изображение;
2. счётчик текущего кадра;
3. заголовок карточки или modal title;
4. optional описание;
5. optional CTA карточки.

### 17.4 Почему это включаем сразу

1. для кейсов и портфолио это один из главных пользовательских сценариев;
2. для каталога работ именно modal делает блок визуально сильным;
3. если не заложить это в family-spec сейчас, потом придётся ломать contract и runtime.

## 18. CTA модель v1

Так как это `commerce-lite`, CTA должен быть отдельной моделью.

Поддерживаемые типы v1:

1. `url`
2. `whatsapp`
3. `telegram`
4. `phone`
5. `none`

Правило:

1. карточка может вести на detail page;
2. primary action может при этом вести сразу в messenger;
3. блок не занимается отправкой формы заказа и не хранит order state.

## 19. Что сознательно не входит в v1

Чтобы линейка не расползлась, в v1 сознательно не делаем:

1. checkout;
2. корзину;
3. оплату;
4. вариации товара;
5. сравнение;
6. избранное;
7. backend faceted search;
8. remote pagination;
9. nested category tree UI;
10. mixed manual + dynamic items в одном и том же списке;
11. giant management UI для hundreds/thousands of products;
12. сложный zoom/pinch engine уровня отдельного gallery app.

## 20. Почему это решение сильное

Такой подход даёт сразу несколько плюсов:

1. продукт остаётся landing-friendly и не превращается в half-baked ecommerce;
2. сохраняется общий NordicBlocks подход: SSR first, manual + data, preview/live parity;
3. появляется новый сильный family-level сценарий для реальных коммерческих сайтов;
4. можно обслужить и портфолио, и услуги, и мини-магазин одной архитектурой;
5. fullscreen media modal со swipe делает линию сильной и для portfolio/cases сценариев;
6. следующие каталожные блоки будут строиться уже поверх общего catalog language, а не с нуля.

## 21. Практический порядок запуска

Для запуска линейки действуем так:

1. сначала утверждаем этот family spec;
2. затем фиксируем первый block spec для `Каталог`;
3. затем проектируем shared catalog engine и contract shape;
4. только после этого идём в код первого block type;
5. затем уже строим `Категории и каталог` и `Витрину каталога` поверх той же базы.

Первый implementation-grade spec для этой линейки вынесен отдельно:

1. `docs/nordicblocks/CATALOG-BROWSER-V1.md`

Итоговая позиция:

это действительно стоит делать не как случайный одиночный блок, а как отдельную сильную каталожную мини-систему внутри NordicBlocks.
